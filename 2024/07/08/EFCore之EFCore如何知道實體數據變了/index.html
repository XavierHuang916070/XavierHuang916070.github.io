  • 執行結果:price與Title均更新

  • 由上述試驗可得知，EF Core會跟蹤實體物件其屬性值的變化

  • 實體類沒有實現屬性值變化的通知機制，EF Core是如何跟蹤到變化的? 其實EF Core預設是透過所謂的<快照更改跟蹤機制>來跟蹤實體物件其屬性值的變化

  • 快照更改跟蹤機制:當DbContext跟蹤一個實體類物件時(例如:透過資料庫實體關聯之DbContext DbSet屬性進行查詢後所返回的物件預設會被跟蹤)，EF Core會創建與實體類對應之資料庫實體的當前快照，當執行到SaveChanges()、SaveChangesAsync()時，EF Core會將當前快照中的值與當前欲變化的實體類物件之屬性值、資料(記錄)進行比較，若兩者相同，則資料庫實體不會有變化，若兩者不同，則資料庫實體會有變化

  • 實體類物件的跟蹤狀態

  • 1 - 已添加(Added):欲將實體類物件進行添加但尚未SaveChange()、SaveChangesAsync()

  • 2 - 未改變(Unchanged) - DbContext正在跟蹤實體類物件且實體類物件之屬性值、資料(記錄)與快照所記錄之資料庫實體的欄位值、資料(記錄)相同，則在呼叫SaveChange()、SaveChangesAsync()時資料庫實體不會有變化

  • 3 - 已修改(Modified) - DbContext正在跟蹤實體類物件且實體類物件之屬性值有修改需求而與快照所記錄之資料庫實體的欄位值不同，則在呼叫SaveChange()、SaveChangesAsync()時資料庫實體中的欄位值會有變化(修改)

  • 4 - 已刪除(Deleted) - DbContext正在跟蹤實體類物件且實體類物件之資料(記錄)有刪除需求而與快照所記錄之資料庫實體的資料(記錄)不同，則在呼叫SaveChange()、SaveChangesAsync()時資料庫實體中的資料(記錄)會有變化(刪除)

  • 5 - 已分離(Detached) - DbContext未跟蹤實體類物件(如下說明)

  • SaveChanges()、SaveChangesAsync()

  • 1 - 未改變(Unchanged)、已分離(Detached)的實體類物件會被SaveChanges()、SaveChangesAsync()忽略

  • 2 - 已添加(Added)的實體類物件，SaveChanges()、SaveChangesAsync()會將其資料(記錄)插入到資料庫實體

  • 3 - 已修改(Modified)的實體類物件，SaveChanges()、SaveChangesAsync()會將其屬性值更新到資料庫實體

  • 4 - 已刪除(Deleted)的實體類物件，SaveChanges()、SaveChangesAsync()會將其資料(記錄)所對應到的資料庫資料(記錄)從資料庫實體中刪除

  • EntityEntry類

  • 1 - 使用DbContext的Entry()可取得資料庫實體所對應到的實體類物件之跟蹤狀態物件

  • 2 - 透過EntityEntry類的State屬性可取得實體類物件的跟蹤狀態

  • 3 - 透過EntityEntry類的DebugView.LongView屬性可取得實體類物件的跟蹤狀態、變化訊息

  • 結論:DbContext會根據跟蹤的實體類物件之狀態，且在SaveChanges()、SaveChangesAsync()時，EF Core核心會根據實體類物件之狀態來生成Insert、Delete、Update的相關抽象語法樹(AST)(此抽象語法樹，不管是針對何種資料庫系統所設計的EF Core Provider都能解析)，之後再透過EF Core Sql Server Provider將抽象語法樹(AST)解析為ADO.NET Core SQL語句，之後再透過ADO.NET Core將ADO.NET Core SQL語句解析為SQL Server SQL語句進而讓SQL Server伺服器執行SQL Server SQL語句
